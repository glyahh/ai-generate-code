# 流式工具调用卡片缺失问题复盘

## 背景

在 Vue 项目首轮对话生成过程中，前端聊天区长期只出现“`[选择工具] 写入文件`”提示，缺少预期的“`[工具调用] 写入文件 ...```内容``` `”卡片。历史回放在刷新后又能看到完整工具输出，说明最终结果入库正常，但实时流式体验异常。

## 现象拆解

- 实时阶段：只见工具请求提示，不见工具执行内容。
- 长时间后刷新：历史消息能重建完整文件卡片。
- 后端构建与产物正常：代码文件写入、`npm build` 可成功。

这个组合现象说明：问题更可能在**流式事件时序**，而非文件写入或历史回放本身。

## 排查思路（按假设推进）

### 假设 1：前端解析器丢包或状态机卡住

检查 `AppChatView.vue` 的 `processAssistantChunkIntoUiState`，确认：
- 能识别 `[选择工具]`、`[工具调用]` 与代码围栏；
- 历史整段文本回放可正确重建卡片。

结论：前端解析器并非根因，更多是上游没有及时给到 `tool_executed` 内容。

### 假设 2：SSE 发送链路被缓冲或阻塞

检查控制器与流处理链路：
- `ChatToGenCodeController` 按 chunk 映射 `ServerSentEvent`；
- `JsonMessageStreamHandler` 处理后继续下游输出。

结论：SSE 通道可持续推送，早期工具请求能准时送达。

### 假设 3：模型侧流式工具调用特性导致“请求早、执行晚”

重点观察 `TokenStream` 回调行为：
- `onPartialToolExecutionRequest` 很早开始且数量巨大；
- `onToolExecuted` 明显滞后，且常在较晚阶段批量到达。

结论：实时卡片缺失的核心在于**执行事件晚到**，不是写文件失败。

## 关键发现

系统原始链路把“文件内容卡片”的出现严格绑定在 `onToolExecuted`：

1. `onPartialToolExecutionRequest` -> 仅输出“选择工具”；
2. `onToolExecuted` -> 才输出“工具调用 + 文件内容”。

当模型供应侧把 `onToolExecuted` 延迟几十秒后，前端就会长期停留在“选择工具”状态。

## 修复方案（最小侵入）

在 `AiCodeGeneratorFacade.turnTokenStreamToFlex()` 增加提前合成策略：

1. 以 `toolCallId` 为键，累积 `onPartialToolExecutionRequest.arguments` 分片；
2. 当累积内容可解析出完整 `writeFile` 参数（`relativeFilePath` + `content`）时，主动合成一条 `type=tool_executed` 消息下发；
3. 真实 `onToolExecuted` 后续到达时，如果同 `toolCallId` 已经合成过，则抑制重复输出，避免双卡片。

该方案保持前端渲染协议不变，只在后端流式适配层补上“提前可用事件”。

## 为什么之前不行，为什么现在可以

### 之前不行

- 卡片渲染依赖真实 `onToolExecuted`。
- 真实回调延迟导致实时阶段缺失关键内容块。

### 现在可以

- 关键参数一旦在 `tool_request` 流中拼完整，就提前生成等价 `tool_executed`。
- 前端立即收到既有协议可识别的数据结构并渲染文件卡片。
- 真实回调到达时通过去重避免重复显示。

## 影响范围评估

- 改动点集中在 `AiCodeGeneratorFacade` 的流式适配函数；
- 不改变数据库结构、不改前端协议、不改工具实现；
- 对非 `writeFile` 工具保持原路径，风险可控。

## 方法论沉淀

这次问题适合采用“事件时序优先”的排查框架：

1. 先定位**哪一类事件缺失**，而不是先改渲染；
2. 区分“实时流问题”和“历史重放问题”；
3. 在适配层做协议等价补偿，优先最小侵入修复；
4. 对补偿消息做幂等去重，防止双写副作用。

该框架可复用于所有“上游延迟、下游等待关键事件”的流式系统故障。
