# 会话记忆指南（会话记忆重构 V4 学习复盘）

面向目标：让你能用**更少的上下文成本**，让模型持续“知道项目进展”，并且在长对话里依然能稳定做**增量修改 / 定位问题 / 继续迭代**。

> 一句话总览：**MySQL 是“真相源账本”**，**Redis 是“热缓存加速”**，**磁盘是“真实代码文件”**；记忆里只放：**摘要（summary）+ 索引（index）+ 短 buffer**，超长内容用 **ref 引用化**。

---

## 核心专有名词（小白版）

### 摘要（summary）
**是什么**：把很多轮对话/改动，用几句话压缩成“到目前为止发生了什么”。  
**为什么需要**：对话很长时，不可能把所有历史原文都喂给模型；摘要能把“剧情”保留下来。

**例子**

> “项目是 Vue3+Vite；已实现登录页；当前在修复按钮点击无响应，怀疑是路由未注册。”

**常见内容要点**
- 当前项目类型/技术栈（Vue/HTML/Multi-file）
- 已完成的功能点
- 当前卡点/待解决问题
- 下一步计划（1~3 条即可）

---

### 索引（index）
**是什么**：不放内容本体，只放“去哪里找”。通常是文件清单/路径/关键 refId/changedFiles。  
**为什么需要**：模型需要“知道该看哪里”，但不需要把每个文件全文塞进上下文。

**例子**

> “本轮改动文件：`src/router/index.ts`、`src/pages/Login.vue`；入口文件：`src/main.ts`；相关 ref：`ref_123`（登录页完整代码）”

**常见内容要点**
- entryFiles（入口/配置/路由等优先级最高的文件）
- changedFiles（本轮真正变更过的文件）
- refs（超长附件的引用列表：refId + 一句话说明）

---

### 短 buffer（short buffer）
**是什么**：最近几轮“原话/小片段”的短缓存（少量原文），用于保持上下文连贯。  
**为什么需要**：摘要会丢细节；短 buffer 保留“最近正在聊的那点细节”，让模型接得上话。

**例子**
- 保留最近 2~6 条 user/ai 的短消息（每条只留 200~800 字）
- 或保留“本轮报错堆栈前 2000 字”

**短 buffer 通常放哪些（你问的重点）**
- 最近用户原话（最近 2~6 条）
- 最近 AI 的关键结论/操作步骤（最近 2~6 条）
- 最近关键证据（报错堆栈/关键日志/接口返回/关键配置片段，截断后）

---

### 引用（ref / 引用化）
**是什么**：当内容太长（整文件、很多文件拼接、大段日志）时，不放进记忆/不直接拼进 prompt，而是单独存成“附件”，用 `refId` 代替。  
**为什么需要**：避免上下文爆炸、避免 Redis/Memory 变得巨大、避免每轮重复塞相同大段代码。

**你可以把 ref 理解成**：“我不把附件全文塞进聊天框，但我会给你一个附件编号；你要看再点开。”

**ref 典型包含**
- refId（唯一编号）
- appId / roundId（属于哪次对话轮次）
- type（代码/日志/文件片段/整文件）
- summary（这份 ref 是什么）
- content（正文，通常存 MySQL；Redis 只做短 TTL 热缓存）

---

### manifest（目录清单索引）
**是什么**：一轮生成完成后，对输出目录做一次“清单记录”，但**不存文件全文**，只存：
- `path` 文件路径
- `hash` 文件指纹（用于对比是否变化）
- `size` 文件大小
- `mtime` 最后修改时间
- `lang` 文件语言（可选）

**为什么需要**：用于计算 `changedFiles`（哪些文件变了），并支撑“按需读文件”策略。

---

### 快照（snapshot）
**是什么**：更偏“概念词”，表示“这一轮结束时的工程状态”。落地时常用 `snapshotId + manifest` 来代表这次快照。  
**注意**：快照不等于把所有文件内容复制一份到数据库；更常见是“磁盘存真实文件 + DB 存索引（manifest）”。

---

### changedFiles（变更文件集合）
**是什么**：把“上一轮 manifest”和“本轮 manifest”对比得到的变化列表。  
**为什么需要**：让模型优先关注“真的变了的文件”，而不是全项目乱读。

---

### inject 文件头（直读磁盘）
**是什么**：新建 AI Service 时，`readFilePageHead` 从 `temp/code_output/...` 读取候选文件并截断前约 10000 字符，以 `[memory_inject]` 写入 ChatMemory。  
**说明**：曾用 Redis `cm:page:*` 缓存该片段，已移除；注入始终以磁盘为准，避免陈旧缓存。

## 一页式流程（读完就能复述）

```mermaid
flowchart TD
  U[用户发一句话] --> DB1[写入 MySQL: chat_history(User)\n拿到 roundId]
  DB1 --> SSE[SSE 流式返回 AI 输出（主链路）]
  SSE --> END{流结束：成功/失败/中断}
  END --> HOOK[doFinally 触发一次\nonRoundCompleted(roundId)]

  HOOK --> STABLE[输出目录稳定性检查\n文件数/mtime 稳定，短重试]
  STABLE --> MANI[生成 manifest（目录清单索引）]
  MANI --> DIFF[与上一轮对比得到 changedFiles]
  DIFF --> MEM[更新 memory_state：summary + index + short buffer]
  MEM --> REF{有超长内容?}
  REF -- 是 --> ARCH[引用化归档 ref\nDB 真相源 + Redis 热缓存]
  REF -- 否 --> OBS[记录日志/指标]
  ARCH --> OBS[记录日志/指标]

  OBS --> CLEAN[后台定时清理 ref/快照\n按天数/条数/容量回收]
```

## “摘要 / 索引 / 短 buffer”分别放什么（建议模板）

### summary（建议 3~8 行）
- 技术栈/模式：……
- 已完成：……
- 当前问题：……
- 下一步：……

### index（建议 5~20 行）
- entryFiles：`...`
- changedFiles：`...`
- refs：`ref_...`（一句话说明）

### short buffer（建议 2~6 段）
- 最近用户原话：……
- 最近关键错误：……
- 最近模型结论：……

## 常见误区（避免踩坑）
- **误区 1：把整坨代码塞进记忆**  
  结果：上下文爆炸、每轮都重复、容易超 token、成本高且不稳定。正确做法：**manifest + changedFiles + ref**。

- **误区 2：把 Redis 当真相源**  
  结果：TTL 一过信息没了。正确做法：**MySQL 真相源**，Redis 只缓存可丢数据。

- **误区 3：收尾失败就让 SSE 断流**  
  正确做法：收尾（manifest/ref/记忆更新）要 **try/catch 自吞**，只打日志/指标，不影响主链路。

---

## 复盘用的“口头背诵版”
**记忆只存三件事：摘要（发生了什么）+ 索引（去哪找）+ 短 buffer（最近聊啥）。**  
**超长内容不塞记忆，变成 ref 附件；项目状态用 manifest 做索引；MySQL 负责真相源，Redis 只是热缓存。**
