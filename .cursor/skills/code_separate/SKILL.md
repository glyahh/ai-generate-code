# code_separate

## 触发

- `/code_separate`、`$code_separate`
- 「解耦 / 抽离 / 拆分 / 单独成类或文件 / 降低耦合」
- 用户指定源文件 + 抽离范围（类、方法块、职责描述）

未显式触发时，仅在用户明确要做**结构性解耦抽离**时使用；普通 bug 修复或小改不要套用本 skill。

## 目标

在理解项目分层与依赖方向的前提下，把用户指定的逻辑拆到合适的新位置，使职责更清晰、依赖更单向，且**用户亲手写的代码文本不被改写**。

## 硬约束（违反即失败）

1. **禁止脚本化重构**：不得用 Python/Shell/codemod/批量替换脚本移动或改写代码；只允许编辑器式**手工增删**（新建文件、复制块、删除旧位置重复块、改调用处）。
2. **用户代码不可改**：用户已存在的语句/方法体/注释，迁到新文件时必须**逐字复制**（允许仅因包名/类名拆分产生的 `package`/`import` 行变化）；禁止重命名用户标识符、禁止重排用户逻辑、禁止「顺手优化」用户实现。
3. **改动范围**：只做解耦所需的新增类/接口/文件 + 从源处删除已迁出块 + **调用侧** import/注入/委托调整；不做无关格式化、不顺手改业务语义。
4. **先读后动**：未完成 Step 1 架构扫描前，不得开始抽离。

## 工作流

### Step 1 — 架构扫描（只读）

以**用户指定的源文件**为锚点：

1. 列出该文件所在包/模块目录下的子目录与同级关键文件（`Glob` / 目录浏览，优先同 feature 域）。
2. 阅读仓库根目录 **`CLAUDE.md`**（存在则必读），对齐：
   - 分层：`controller → service → serviceImpl → mapper`；`core` 门面 + Parser/Saver 执行器模式
   - 生成链路：`AiCodeGeneratorFacade` / `WorkflowCodeGeneratorFacade` 与传统 vs 工作流路径
   - 新增类型扩展点：`CodeGenTypeEnum` + Parser + Saver 注册
3. 若涉及后端 Java，可对照 `.cursor/skills/backend-design/SKILL.md` 的分层与响应/异常约定（只读，不扩大改动面）。
4. 输出 ≤15 行「扫描摘要」：当前类职责、主要依赖、建议依赖方向、拟拆边界。

### Step 2 — 澄清抽离意图（信息不足时）

用 1–3 个具体问题收敛（避免一次问太多）：

- 要拆出的**具体范围**（哪些方法/哪段逻辑/哪类职责）？
- 新单元命名与放置包（是否有既有 `util`/`helper`/`support` 惯例）？
- 是否允许新增接口、是否必须保持原 public API 不变？
- 验收标准：编译通过 / 特定测试 / 行为零变化？

用户已在 prompt 中说清的，不再重复追问。

### Step 3 — 抽离方案（先方案后动手）

给出简短方案（用户未要求可省略展示，但内部必须遵守）：

| 项 | 内容 |
|----|------|
| 迁出块 | 方法/内部类/字段列表（引用源行号） |
| 新建物 | 新类/接口/文件名与包路径 |
| 依赖规则 | 新类依赖谁；谁依赖新类（避免环依赖） |
| 用户代码处理 | 标明「原样复制」的块 |
| 调用调整 | 哪些文件仅改 wiring（注入、静态调用、import） |

**glyahh 常用落点**（按职责选，勿堆上帝类）：

- 纯算法/无状态：`core/util`、`.../support`
- 解析/落盘：`*Parser`、`*Saver` + 执行器注册
- 流式/SSE 片段：`core/handler` 或现有 `StreamHandler*` 旁路
- AI 调用封装：留在 `ai/` 工厂邻域，不把 Web 层逻辑下沉到 `core`
- 工作流节点： `LangGraph4j/node` + `state`，不塞进 Facade 巨型方法

细节模式见 `.agents/skills/code_separate/references/glyahh-decouple.md`（仅在与本仓库相关时读取）。

### Step 4 — 执行抽离（仅增删 + 原样迁移）

顺序：

1. **新建**目标文件/类/接口，将用户代码块**原样粘贴**进去（仅补必需的 `package`、类壳、访问修饰符）。
2. 在源文件中**删除**已迁出的重复块（若整段迁移）；若保留委托入口，委托方法体只做一行转发，不改写用户原实现。
3. **禁止**在迁移过程中修改用户块内部的任何一行。

### Step 5 — 优化调用侧（允许改动）

仅调整「接线」代码，使用户/系统其余部分仍按原行为工作：

- 更新 `import`、`@Resource`/`构造器注入`、静态工具类引用
- 原大类改为委托：`private final XxxSupport support` + 方法转发
- 必要时在**非用户编写**的胶水层补充适配方法（薄包装，无业务新逻辑）

仍须遵守：不改用户已写方法体内部实现。

### Step 5.1 — 抽离后在调用侧补注释（推荐）

逻辑迁到 `*Support` / `*Util` 等静态工具类后，在**原 Service / Facade** 里，每一处 `XxxSupport.method(...)` 调用的**上一行**加一句短注释，说明这一步在业务流程里干什么（不是把方法名翻译成中文）。

- **一行即可**：写清动作与作用对象，例如「按列宽截断正文，防止超长插入失败」。
- **少套话**：避免「得到…」「用于…的…」「确保…」等 AI 腔（可与 `/humanizer` 联用润色注释）。
- **与步骤注释并存**：已有 `// 1.` 流程注释时，可在 support 调用前再补半句，专指该工具方法。
- **委托入口**：一行 `return Support.foo(...)` 的薄包装，在 `return` 上一行注明「转发到 support」及用途（如单测反射入口）。
- **不改 support 内部**：Step 6 前不在 Support 内加注释；迁出块内的原有注释仍原样保留。

### Step 6 — 新建 Support 类注释（`/comment`，默认必做）

抽离完成后，对**本次新建**的 `*Support` / `*Helper` 类执行 **`/comment` skill**（读 `~/.cursor/skills/comment/SKILL.md` 或仓库镜像）。

| 规则 | 说明 |
|------|------|
| 范围 | **仅**新建 Support 文件；不回头改源 ServiceImpl（除非用户另要求） |
| 已有注释 | 自源类原样迁来的 Javadoc / `//` **原文保留**，不覆盖、不重写首句 |
| 补缺 | 无 Javadoc 的方法补全块注释；已有块但缺 `@param` / `@return` 的**只补缺失标签行** |
| 类头 | 非配置类补 `a -> b -> c` 主链路（Step 4 已写且够用可不再改） |
| 禁止 | 借注释之名改方法体、改签名、改用户迁移代码逻辑 |

用户明确说「不要注释」时，可跳过本步。

### Step 7 — 验证与交付

1. 运行与变更相关的最小验证（如 `./mvnw -q -DskipTests compile` 或用户指定测试）；前端抽离则 `npm run build` / `lint` 仅当本次触及前端。
2. 交付物：
   - 变更摘要（拆出什么、依赖方向）
   - 文件清单（新增/修改/删除）
   - 标明「用户代码原样保留」的块
   - Support 类注释补全情况（哪些方法新增/补 `@param`）
   - 剩余风险（循环依赖、线程安全、Bean 作用域）

## 禁止项

- 脚本批量移动/重命名/格式化
- 修改用户已有方法体、条件、日志文案、魔法数
- 无关重构（换库、改异常体系、改 API 契约）除非用户明确要求
- 用「重写实现」代替「复制迁移」
