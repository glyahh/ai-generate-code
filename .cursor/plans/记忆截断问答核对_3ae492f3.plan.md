# 记忆与截断：对你 7 个问题的代码级答复

## 1. 一次性超长 HTML：会不会 MySQL 不完整、前端不完整、AI「完全缺一块」？

分三条链路看（不要混成一条）：

**入库（MySQL）**  
[`ChatHistoryServiceImpl.addChatMessage`](d:\mainJava\all Code\program\glyahh-ai-generate-code\src\main\java\com\dbts\glyahhaigeneratecode\service\impl\ChatHistoryServiceImpl.java) 使用 `sanitizeMessageForPersistence`，按 `information_schema` 里 `message` 列的 `CHARACTER_MAXIMUM_LENGTH` 截断并追加 `...[message truncated]`。  
同一方法里的 `ensureAuditColumnsIfMissing` 会在缺列或类型不是 `longtext` 时执行 `MODIFY COLUMN message LONGTEXT`（见约 134–147 行）。  
仓库 [`sql/create_table.sql`](d:\mainJava\all Code\program\glyahh-ai-generate-code\sql\create_table.sql) 里仍是 `message text`（约 65KB 量级），**若从未跑过会触发该迁移的入库路径**，老库确实可能截断；迁移成功后 LONGTEXT 下正常单次 HTML 一般不会因列类型被截断。

**给模型的上下文（Redis / ChatMemory）**  
这与 `compactAiMessageForMemory` 有关：**不发生在 `addChatMessage` 写库时**，发生在 (a) `turnHistoryToMemory` 把 DB 历史灌进 Redis 时，对 HTML/MULTI_FILE 且长度 `> MEMORY_AI_MESSAGE_MAX_LENGTH`（**2400**，见约 79、584–617 行）的 **AI** 条做摘要；(b) `compactMemoryMessagesIfNeeded` 在线压缩 Redis 里已有 AI 条（约 908–947 行）。  
注释写明：**灌 Redis 时压缩，不改 DB 原文**（约 578–582 行）。

**结论**  
- 可能出现：**DB 里很长、列表/详情里也能看到很长**，但 **下一轮生成时模型看到的 Redis 里已被压成片段**（>2400 的 AI 条），观感像「忘了中间一大段」，**不等于** MySQL 一定缺（LONGTEXT 且未超上限时库可以是全的）。  
- 若列仍是 `TEXT` 且未升级：**会出现**库不完整 + 回显不完整 + 后续从库重建也不完整。

## 2. `maxMessages(80)` 会不会「永远不可能触发」？和「最多 3 轮」压缩的关系？

当前仓库里：

- [`MessageWindowChatMemory`](d:\mainJava\all Code\program\glyahh-ai-generate-code\src\main\java\com\dbts\glyahhaigeneratecode\ai\aiCodeGeneratorServiceFactory.java) 使用 **`.maxMessages(80)`**（约 155–160 行）。
- [`ChatHistoryConstant.MAX_ROUNDS_BEFORE_SUMMARY`](d:\mainJava\all Code\program\glyahh-ai-generate-code\src\main\java\com\dbts\glyahhaigeneratecode\constant\ChatHistoryConstant.java) 在仓库中为 **3**；`CHAT_MEMORY_MAX_MESSAGES` / `MEMORY_PRELOAD_MESSAGE_ROWS` 与 `aiCodeGeneratorServiceFactory` 应对齐（见「会话记忆压缩改造」计划）。

**轮数压缩**（`trySummarizeOldestRoundsIfNeeded`）统计的是 **Redis 里 `UserMessage` 条数**（`countRoundsByAppIdInternal`），超过 8 才合并（约 736–748 行）。**它不保证** Redis 里总消息条数 ≤ 某值：工具调用、多段 AI、系统类消息等都会占 `maxMessages` 额度。

**预加载条数**：`turnHistoryToMemory(appId, build, 20)` 只从 DB 拉 **最近 20 条**记录（约 500 行 `limit(1, maxCount)`），不是「全历史」。

**结论**  
- 「永远不可能触发 80」**不成立**：长会话 + 工具链 + 多段消息时仍可能顶到窗口上限。  
- 若你本地/分支已把策略改成「等价于只保留 3 轮」，需要以**你分支实际常量与调用链**为准；**本仓库快照**是阈值 **8** 轮触发 Redis 合并，且 `maxMessages(80)` 与常量注释不一致。

## 3. `trySummarizeOldestRoundsIfNeeded` 没看懂、没用就删掉？

**在用**：[`ChatToGenCodeImpl.chatToGenCode`](d:\mainJava\all Code\program\glyahh-ai-generate-code\src\main\java\com\dbts\glyahhaigeneratecode\service\impl\ChatToGenCodeImpl.java) 约 108–110 行（普通入口）；`chatToGenCodeByWorkflow` 约 159–160 行（条件调用）。  

**行为**：当 Redis 统计的用户轮数 **> 8** 时循环：从 DB 取最早 4 条（2 轮）→ 调模型生成摘要 → **`syncRedisOnly`**：用摘要替换 **Redis 列表头部 4 条为 2 条**（约 886–904 行）。**写回 DB 的合并代码整块被注释掉**（约 770–787 行），因此 **DB 仍保留原始多行**，与 Redis 可能长期不一致。

**结论**  
- **不是死代码**；删掉会改变 **>8 轮** 时的 Redis 上下文行为。  
- 若产品上要「DB 与 Redis 一致」或「永不摘要」，需要**改策略**而不是简单删方法（否则长会话 Redis 膨胀、token 与存储压力回来）。

## 4. `message` 列「这道不会吧」？

**运行时**会尽量把 `message` 升到 **LONGTEXT**（见 `ensureAuditColumnsIfMissing`）。  
**但若**线上库从未走过该路径、或权限不足导致 `ALTER` 失败、或仍使用极老的 `TEXT` 且 `CHARACTER_MAXIMUM_LENGTH` 为 65535，`sanitizeMessageForPersistence` **仍会**按上限截断。

**结论**：「默认不会」取决于线上是否已是 LONGTEXT 且迁移成功；**不能从语义上假设永远不发生**，应用信息_schema或运维规范确认一次最稳。

## 5. Redis TTL 过期：你说能从 MySQL 回显？

**对一半**。  
[`aiCodeGeneratorServiceFactory`](d:\mainJava\all Code\program\glyahh-ai-generate-code\src\main\java\com\dbts\glyahhaigeneratecode\ai\aiCodeGeneratorServiceFactory.java) 在缓存命中时若发现 Redis 空，会 **invalidate Caffeine** 并重建 service，随后 `turnHistoryToMemory` 从 **MySQL 最近 20 条** 再灌 Redis（约 111–118、165–168 行）。

**但**：灌入时仍会对 HTML/MULTI_FILE 的长 **AI** 条执行 `compactAiMessageForMemory`（`turnHistoryToMemory` 约 559–561 行）。所以：**聊天列表回显可以长**，**下一轮模型上下文仍可能短**。

## 6. `CodeQualityCheckNode` 的 512KB / 160000 字符：会不会因「只有几轮对话」就永远不触发？

**不依赖对话轮数**。质检读的是 **生成目录下的文件**（见 [`CodeQualityCheckNode`](d:\mainJava\all Code\program\glyahh-ai-generate-code\src\main\java\com\dbts\glyahhaigeneratecode\LangGraph4j\node\CodeQualityCheckNode.java) 约 48–57 行：`MAX_FILE_BYTES`、`MAX_PROMPT_CHARS` 是 **字节/字符上限**，与 Redis 里剩几轮无关。  
单文件 >512KB 或拼接后 >160000 **字符**（不是 token）就会截断参与质检的上下文。

「前端提示上下文已满」属于**产品化另做**：当前片段未与 `CodeQualityCheckNode` 绑定，需要单独设计（例如质检前统计、或捕获超长后 SSE 错误文案）。

## 7. `HtmlMultiFileEditContextBuilder` 片段上下文：是幻觉吗？有什么约束方案？

不完全是「幻觉」一词能概括：本质是 **模型可见上下文 ≠ 磁盘全文** 时，模型更容易 **整文件重写** 或 **漏改/多改**。

**工程侧可落地的约束（与现有「工具改盘」方向一致）**  

- **强约束**：编辑类意图只允许 `readFile` + `modifyFile`（或 patch），禁止 `writeFile` 覆盖整文件；在 tool 层或 prompt 层写死。  
- **协议层**：`modifyFile` 要求 **唯一锚点**（或限制 `replace` 命中次数为 1），避免一段 `oldContent` 多处匹配导致大面积替换。  
- **验证层**：改后跑 HTML 语法检查、关键选择器存在性检查、或轻量 E2E（你们已有 Vue 语法工具链，可对 HTML/MULTI 做最小校验）。  
- **上下文层**：编辑轮 **显式注入**「当前文件头尾 + 变更区」或 hash，而不是只靠对话里的旧 AI 消息（避免与 2400 压缩打架）。
