# 修复工作流重试与慢提示

## 背景结论
- 已定位到“总结 HTML/CSS/JS 再次重试”的根因在后端历史记忆压缩逻辑：[`src/main/java/com/dbts/glyahhaigeneratecode/service/impl/ChatHistoryServiceImpl.java`](src/main/java/com/dbts/glyahhaigeneratecode/service/impl/ChatHistoryServiceImpl.java) 会在请求前压缩超长 AI 历史，并对超长代码块调用 `summarizeLongCodeBlock()` 触发额外模型请求，把“代码用途总结”写回上下文。
- 该压缩在缓存命中时被主动触发：[`src/main/java/com/dbts/glyahhaigeneratecode/ai/aiCodeGeneratorServiceFactory.java`](src/main/java/com/dbts/glyahhaigeneratecode/ai/aiCodeGeneratorServiceFactory.java) 的 `getAiCodeGeneratorService()` 中调用 `compactMemoryMessagesIfNeeded()`。
- 工作流重试路径本身存在：[`src/main/java/com/dbts/glyahhaigeneratecode/LangGraph4j/CodeGenWorkflow.java`](src/main/java/com/dbts/glyahhaigeneratecode/LangGraph4j/CodeGenWorkflow.java) 在 `qualityResult.isValid=false` 时走 `retry -> code_generator`，因此“额外总结+质检失败”会放大为循环感。
- 你新增日志已验证时序：第 1 次质检 `isValid=false` 后触发重试；重试轮命中缓存时执行 `compactMemoryMessagesIfNeeded()`，立刻出现 3 次“总结 HTML/CSS/JS”模型调用；随后再次生成并质检通过（`isValid=true`）结束。说明问题是“重试轮的记忆压缩策略”而非工作流拓扑错误。
- 前端工作流卡片渲染位置已确认：[`ai-generate-code-frontend/src/page/App/AppChatView.vue`](ai-generate-code-frontend/src/page/App/AppChatView.vue) 与 [`ai-generate-code-frontend/src/utils/workflowChatFilters.ts`](ai-generate-code-frontend/src/utils/workflowChatFilters.ts)。

## 变更边界（按你的约束）
- 以**最小范围修改**为原则，优先在 workflow 专属节点、专属策略类、局部装配点处理问题。
- **不改老代码文件**与**高复用核心逻辑**（除非只做一行级别的安全开关注入且可回滚）。
- 采用高内聚低耦合方式：新增小型策略/配置对象或局部 helper，避免改动全局公共流程。
- 不改变既有核心业务语义（工作流拓扑、通用生成主链、公共工具契约）。

## 核心主策略（最高优先级）
- **失败轮不总结**：代码质检失败后，禁止触发任何 AI 总结压缩动作。
- **定向清理失败产物**：仅删除 Redis 中“本次失败代码生成对应的 AI 长消息”，不清空整段会话记忆。
- **成功后再压缩**：仅在当前轮质检通过时，才允许执行一次压缩（且优先轻量截断，不依赖 AI 总结）。
- **目标**：同时满足“去长等待 + 降 token + 不丢用户意图”。

## 实施方案
1. **后端：把“代码块 AI 总结”从请求前链路移除，仅保留安全截断**
   - 在 workflow 专属调用路径新增“压缩触发开关/策略”，避免直接改动公共历史压缩主逻辑。
   - 在 `ChatHistoryServiceImpl` 中仅做局部注入点改造：压缩不再在请求前和缓存命中时触发，避免进入 workflow 重试回路。
   - 压缩执行时机改为：**仅当当前轮 `CodeQualityCheckNode` 判定通过后**，再对 Redis 记忆做一次压缩。
   - 对超长代码块采用固定长度截断（保留语言围栏和可读前缀），去掉每轮都调用模型做“代码总结”的路径。
   - **失败轮（质检未通过）完全禁用 AI 总结压缩**，避免对失败代码做无意义总结（主策略）。

2. **后端：限制“压缩记忆”触发时机，避免重试轮污染**
   - 在 `aiCodeGeneratorServiceFactory` 通过局部条件分支屏蔽 workflow 重试场景的压缩触发，不重构其公共缓存机制。
   - 在 workflow 结束分支中新增“质检通过后压缩”入口（仅执行一次），保持 `qualityResult=false -> retry` 机制不变。
   - 在 `WorkflowContext` 增加重试计数（`retryCount`），超过 3 次后启用“轻量截断兜底（不调用 AI 总结）”或中止重试，防止 token 失控。
   - 当首轮质检失败时，定向清理 Redis 中“首轮生成代码对应的 AI 长消息记录”（仅删除失败产物，不清空整段会话记忆），再进入重试生成（主策略）。

3. **后端：修复图片拼接缺失（image20 / mermaid）并加入配额**
   - 仅在 workflow 图片收集与提示词增强链路中处理，不改动通用图片工具基础能力。
   - 排查 `ImageCollectorNode` 到 `PromptEnhancerNode` 的资源组装逻辑，修复只拼接 pixel/CONTENT、漏掉 image20 与 mermaid 的问题。
   - 加入明确数量策略：pixel 搜索图最多 15 张；image20 AI 生成图最多 5 张；mermaid 图按需生成且最多 2 张。
   - 若存在提示词长度限制，采用“先保留类别覆盖、再裁剪描述文本”的策略，避免某一类（如 pixel）挤占全部配额。

4. **前端：为“代码生成中...”增加延时提示（你描述的右侧提示）**
   - 在 `AppChatView.vue` 为 workflow 最新步骤卡片增加“长耗时阈值状态”（例如流式开始后 120s）。
   - 当最新步骤为“代码生成中...”且超过阈值时，在该步骤右侧显示提示文案：`Taking longer than expected`（可加中文副文案）。
   - 使用 `ui-ux-pro-max` 风格做轻量高辨识样式：弱强调琥珀色信息徽标、清晰对比度、无布局抖动、移动端可读。

5. **验证与清理**
   - 后端：跑针对 workflow/multi_file 的最小回归（至少一次“质检通过”和一次“质检失败后重试”）。
   - 后端：验证“失败轮不总结 + 定向清理失败代码消息 + 成功后压缩一次”三条主策略按预期生效。
   - 后端：验证增强提示词中出现三类资源（pixel/image20/mermaid）且满足 15/5/2 配额上限。
   - 前端：本地验证延时提示按阈值出现/结束后消失，且不影响历史消息卡片。
   - 清理本次修复过程中的临时日志、一次性调试代码与注释，避免遗留“测试垃圾”。

## 预期结果
- 不再出现“代码质检后触发 HTML/CSS/JS 用途总结，再次生成”的无效链路。
- 压缩行为只在当前轮质检通过后执行一次，不再每轮判断、每轮压缩。
- 质检失败轮不再做任何 AI 总结压缩；重试超过 3 次时仅走轻量截断兜底。
- 首轮质检失败时，Redis 中首轮失败代码消息会被定向清除，避免失败代码持续占用 token。
- workflow 仅在真实代码不通过时重试；通过后直接结束（或按 Vue 分支进入构建）。
- 增强提示词稳定包含 pixel/image20/mermaid 三类图片资源，并满足 15/5/2 的数量策略。
- 用户等待过久时，卡片右侧出现明确且美观的“耗时较长”提示，降低焦虑感。

## 非目标（本次不做）
- 不做“失败后改为局部修复节点（code_fixer）替代全量重生”的架构升级（该项可作为后续优化）。
