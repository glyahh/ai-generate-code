# 修复历史回显异常（精简版）

## 问题结论
- workflow 链路 `TOOL_REQUEST` 缺少等价 `Set<toolId>` 去重，导致同一工具调用可被重复回显。
- `WorkflowTextStreamHandler` 全量拼接 chunk 入库，重复回显会直接写入 `chat_history.message` 并放大。
- 首轮实时显示与历史回放是不同解析路径，历史回放更容易暴露已入库的脏内容。

## 目标
- 对齐 ai service 链路思路：在 workflow chunk 适配阶段基于 `Set<toolId>` 自动去重。
- 减少 workflow 入库噪声，避免 `chat_history.message` 异常膨胀。
- 修复前端历史分页重复合并问题。
- 提供一次性 MySQL + Redis 清理脚本，清理本次脏数据。

## 实施项
1. 后端去重（主修复）
   - 文件：`src/main/java/com/dbts/glyahhaigeneratecode/core/WorkflowCodeGeneratorFacade.java`
   - 做法：在 workflow 流上下文维护 `Set<String> seenToolIds`；`adaptWorkflowCodeChunk(...)` 中 `TOOL_REQUEST` 若 `toolCallId` 已出现则直接忽略。

2. 后端入库控量（兜底）
   - 文件：`src/main/java/com/dbts/glyahhaigeneratecode/core/handler/WorkflowTextStreamHandler.java`
   - 做法：对重复工具展示噪声做轻量过滤，避免无意义重复文本入库；不改 SSE 协议。

3. 前端历史去重（放大器修复）
   - 文件：`ai-generate-code-frontend/src/page/App/AppChatView.vue`
   - 做法：`load more` 合并历史时按 `id` 去重（保留兜底 key），避免重复渲染。

4. 一次性脏数据清理
   - 目录：`Platform-Utils/script/`
   - 内容：MySQL 清理重复/异常超长消息；Redis 清理对应 app chat memory，触发从 MySQL 重建。

5. 回归验证
   - workflow 同一 `toolCallId` 只回显一次。
   - `chat_history.message` 体量恢复正常区间。
   - 前端历史分页不重复，页面不卡顿。

## 执行顺序
- 先后端去重 -> 后端入库控量 -> 前端历史去重 -> 清理脚本 -> 回归验证。
