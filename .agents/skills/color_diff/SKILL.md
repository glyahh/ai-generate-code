# color_diff

## 触发

- `/color_diff`、`$color_diff`
- 用户提供一张或多张带**彩色框/箭头**的截图，并要求按图修改
- 「按标注图改」「红框删掉」「黄框改成…」「拉到绿线」「蓝框优化文案/样式」

未附带标注图且未提及彩框语义时，不要套用本 skill。

## 优先级（强制）

1. **用户当次提示词**（文字说明、文件名、顺序、例外）  
2. **本 skill 的框色/箭头默认语义**  
3. 项目既有规范（如 `frontend-design.md`、`CLAUDE.md`）

冲突时以用户提示词为准，并在交付摘要中注明「覆盖了哪条默认」。

## 标注语义（默认）

| 标记 | 含义 | 允许改动 |
|------|------|----------|
| **红框** | 删除框内文字 / 图片 / 组件 | 移除 DOM/节点、资源引用、对应样式；清理死代码与无用 import |
| **黄框** | 修改框内文字 / 图片 / 组件 | 按用户提示词改内容、替换资源、改 props/文案/样式；**具体改什么以用户说明为准** |
| **绿框** | 布局基准线 | 将某布局容器拓宽/对齐到该基准（常与箭头连用）；改尺寸、grid/flex、margin/padding、列宽等 |
| **箭头** | 聚焦指针 | 标明要操作的**目标**元素；或与绿框配合表示「从当前框拉到基准」 |
| **蓝框** | 逻辑不变的内容优化 | 文案润色、排版、间距、颜色、可访问性、命名可读性等；**不得改变框内业务逻辑、分支、API 契约、数据流** |

同一区域多色重叠或含义不清：**先列出理解表，再动手**（见 Step 2）。

## 工作流

### Step 1 — 读图与建清单（只读）

对每张图：

1. 识别技术上下文（页面 URL、组件名、文件路径若用户已给则记录）。
2. 按框色/箭头枚举条目，建议表格：

| ID | 图序号 | 颜色/箭头 | 图中位置描述 | 默认动作 | 用户补充说明 |
|----|--------|-----------|--------------|----------|--------------|
| 1 | 1 | 红框 | 左上标题 | 删除 | … |

3. 多图时：按用户指定顺序处理；未指定则按图序号 + 同图从上到下、从左到右。
4. 箭头无框时：箭头尖端/指向侧为**操作目标**；箭尾或旁注为**参照**（常见为绿基准）。

### Step 2 — 歧义与阻塞

以下情况**先问 1–3 个具体问题**，不要猜：

- 黄框无改法说明且图中文字不可辨认  
- 红框是否「仅隐藏」还是「彻底删除含逻辑」  
- 绿框基准是 min-width、对齐边还是占满父级  
- 蓝框优化是否允许改接口字段名/枚举值（默认不允许）  
- 多张图对同一区域指令矛盾  

用户已在提示词说清的，不再追问。

### Step 3 — 定位代码

1. 用图中可见文案、组件结构、类名、路由、截图文件名反查源码（`Grep` / 目录浏览 / 打开相关 Vue、React、HTML、CSS 文件）。
2. 一项标注对应尽量**单一文件或单一组件**；跨文件依赖在清单中标注。
3. 找不到实现：报告已搜路径与关键词，请求用户提供文件路径或页面路由。

### Step 4 — 实施（按清单逐条）

执行顺序建议：**红（删）→ 绿+箭头（布局）→ 黄（改）→ 蓝（优化）**，避免先改后删白费；用户指定顺序则从之。

- **红框**：删除节点与专用样式；移除仅服务于该元素的逻辑；不保留「注释掉的尸体代码」除非用户要求。
- **黄框**：严格按用户文字改；未说明时仅做图中可辨认的最小可见差异（如替换占位文案），并在摘要标注「待确认」。
- **绿框 + 箭头**：先确定被移动的容器与基准边；用现有布局体系（flex/grid/width）扩到基准；不引入新 UI 库除非用户同意。
- **蓝框**：仅 presentation / copy / a11y；不改条件分支、不算力、不改请求参数语义。

### Step 5 — 验证

- 前端：在相关页面目视或运行项目既有 lint/build（用户环境允许时）。
- 对照原图检查：删除处无残留、黄框目标已变、绿框对齐、蓝框逻辑路径未变。
- 交付前勾选下方检查表。

## 交付格式（强制）

```markdown
## color_diff 变更摘要

- 依据：图 [1..n] + 用户提示词要点
- 覆盖默认：（无 / 列出）

### 清单执行情况
| ID | 状态 | 文件 | 说明 |
|----|------|------|------|

### 涉及文件
- 修改：…
- 删除：…

### 待确认 / 风险
- …
```

## 检查表

- [ ] 每条标注均有对应处理或明确「跳过+原因」
- [ ] 红框：无残留节点/样式/死引用
- [ ] 黄框：与用户说明一致
- [ ] 绿框/箭头：布局达到基准意图
- [ ] 蓝框：无业务逻辑/API 行为变化
- [ ] 用户提示词优先项已落实

## 禁止

- 忽略用户文字说明，仅按框色臆测大改
- 蓝框内改逻辑、改接口契约、改数据库字段语义
- 无关文件大范围格式化或「顺手重构」
- 未读图就批量替换全仓库相似字符串

## 与其他 skill

- 纯前端 UI 实现可结合 `/build-frontend`；本 skill 负责**从标注图到变更清单**的主流程。
- 结构性抽离不在此 skill 范围，除非用户明确要求且标注为黄/蓝框内的重构说明。
