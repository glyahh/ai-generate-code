# git_commit

用法：当用户说 `/git_commit`、`$git_commit`、「提交说明 / commit message / PR 描述 / 变更总结」时使用。Cursor、Codex、Claude Code 三端兼容。

## 适用场景

当用户输入 /git_commit（可附带图片/截图）并希望你基于当前工作区的实际修改内容，生成可直接复制的提交说明/PR 描述时使用。

## 核心原则

1. 只信本地 git diff 与仓库状态，不以图片文字为准（图片仅作为“用户意图/范围提示”）。
2. 先读 diff 再写总结，优先覆盖“影响行为变化/并发隔离/稳定性/可扩展性/竞态治理”等高价值点。
3. 输出必须严格满足用户格式约束：固定 6 区块、**空行与缩进必须完全一致**、编号规则严格、句式规则严格、无额外 markdown 装饰。
4. 输出必须包裹在单个 ```text 代码块中，且代码块内部只能是模板正文：不得添加标题解释、表格、分隔线（`---`）、项目符号（`-`）或任何额外内容。
5. 即使用户已给出一段“半成稿文本”（如仅有分区标题但无序号），也必须自动重排为模板格式，并补齐 `前端/后端/其他` 的严格序号（从 `1.` 开始递增）。

## 执行步骤（必须按顺序）

1. 获取变更范围
   - 运行 git status（确认 modified / added / untracked）
   - 运行 git diff --name-only（列出变更文件清单）
   - 运行 git diff（获取完整差异）
   - 如存在 staged 变更，再运行 git diff --staged（补齐差异）

2. 快速归类变更点（按“对用户有价值”排序）
   - 并发/异步/多实例隔离（prototype、线程、SSE 多流、连接生命周期）
   - 配置拆分与装配（config 分层、Factory 化、依赖注入边界）
   - 存储/持久化/缓存（Redis、DB、Caffeine、TTL、重建策略）
   - 工具编排/工作流（tools 组合、白名单、策略路由）
   - 构建/依赖/运行时行为（pom、yml、devtools、timeout）

3. 生成最终输出（一次性输出，无多余解释）
   - 用下面“固定输出模板”
   - 最终答案必须是一个且仅一个 ```text 代码块，代码块外不得出现任何字符（包含“好的/说明/总结”等）
   - 每条内容使用“结论短语: 解释补充”，逗号分隔，句子短，信息密度高
   - 编号规则必须严格执行：在“前端/后端/其他”三个分区内，条目必须从 1 开始按 1、2、3… 递增（不得跳号、不得重复、不得省略序号）；如果该分区确实无改动，也必须输出 `1. 无`
   - 若用户输入的是无序号版本，先按语义映射到对应分区，再输出为带序号版本；禁止原样返回无序号内容
   - 技术词保留英文（SSE、streamId、EventSourceMap、prototype、Factory、ChatModel 等）
   - 尽量体现高级工程语言：并发隔离、竞态治理、资源边界、装配解耦、可扩展性、稳定性、可观测性
   - **严格换行规则**：各区块标题行与内容之间必须保留 1 个空行；区块之间必须保留 2 个空行；序号行必须从行首开始（不得缩进）

4. humanizer 定稿（可选，与 `/humanizer` 协同）
   - 定稿前可阅读 `.agents/skills/humanizer/SKILL.md`，仅润色各条「结论短语: 解释补充」的自然语言
   - **不得**改动 6 区块标题、空行、编号、```text 代码块结构；技术锚点（类名、路径、配置键）原样保留

## 固定输出模板（必须完全一致，包括缩进、空行、序号）

系统优化阶段

目标: <用 -> 串联流程, 保持一致的标点风格>


前端:

1. 无


后端:

1. <结论短语>: <解释补充>

2. <结论短语>: <解释补充>


其他:

1. <结论短语>: <解释补充>


author By glyahh

## 输出前自检（必须执行，避免格式跑偏）

在最终输出前，逐条对照检查：

1. **只能有 6 个区块**：`系统优化阶段`、`目标:`、`前端:`、`后端:`、`其他:`、`author By glyahh`
2. **空行必须一致**：`目标:` 与 `前端:` 之间为 2 个空行；各分区之间为 2 个空行；`author By glyahh` 前为 2 个空行
3. **编号必须存在且从 1 开始**：`前端/后端/其他` 各自独立从 `1.` 起；无内容时也必须输出 `1. 无`
4. **序号行不允许缩进**：必须以 `1.` / `2.` 开头，前面不能有空格或 Tab
5. **用户原文若无序号也必须修正**：不得输出未编号的分区正文
6. **禁止输出任何模板外文本**：禁止“解释/补充说明/额外 markdown 装饰/结尾总结”等
7. **代码块规则必须满足**：必须使用且仅使用一个 ```text 代码块包裹模板正文，代码块外不得出现任何文本

## 生成内容的默认策略（可按 diff 自动调整）

1. 编号必须递增: 前端/后端/其他 三个分区各自独立从 `1.` 开始，逐条递增到 `2.` `3.`…，不得跳号/不得省略序号
2. 无改动也要占位: 若某分区无内容，必须输出 `1. 无`（保持模板结构稳定，便于复制粘贴）
3. 分区至少 1 条: 前端/后端/其他 至少各 1 条（确无内容用 `1. 无` 占位；其他优先写 1 条“开发小趣事/长期 bug/边角治理/彩蛋”，但必须能从 diff 或 git 输出找到锚点）
4. “其他”可活泼但不虚构: 必须能从 diff 或 git 输出找到锚点（例如 LF/CRLF 警告、devtools 冲突、超时/日志开关、目录命名等）
5. 后端优先写 2+ 条: 若涉及模型/配置/Factory/并发等高价值点，优先写 2 条以上
6. 只写结论与影响: 不写“我做了什么/我将要做什么”，用“结论短语: 解释补充”表达
7. 避免夸大: 每条必须可回溯到变更点或可观测现象
