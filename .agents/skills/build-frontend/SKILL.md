## 用法

- 触发：用户显式使用 **`/build-frontend`**
- 覆盖：写新页面/组件、修 bug、UI/UX 打磨、工程化、必要时的抽离重构

## 项目风格与技术栈对齐（强制）

- 默认技术栈：**以仓库前端部分的真实现状为准**；若用户明确指定新技术栈，则按用户指定执行。
- 规范来源：优先读取根目录 **`frontend-design.md`**。
  - 若不存在：**提醒用户创建**该文件后再继续做“对齐风格/抽离目录/工程化”类工作；其它小修小补可继续按现有代码风格最小改动推进。

## 核心原则（强制）

- **修改现有代码（默认场景）**
  - 小问题：最小改动优先，避免无关重构扩大影响面。
  - 只有当出现以下情况，才允许进行抽离/重构：
    - 反复修改仍有问题（同类问题持续出现）
    - 发现底层/关键逻辑存在重大错误或结构性缺陷
  - 若发生抽离/重构：必须说明收益、影响面、风险点与回滚建议。

- **编写新代码（新增场景）**
  - 优先对齐项目既有架构与编码风格，再实现 UI/UX 与功能需求。

## 注释标准（强制）

- 每个函数/方法：在其最前面写清楚**这个方法有什么用**（意图、输入输出、边界/副作用）。
- 关键逻辑：对复杂/易错/有隐含约束的关键逻辑，必须注释说明**意图、坑点与边界**。
- 禁止流水账：不写逐行重复代码含义的注释。

## 抽离与目录规范（强制：Feature-first）

以“大功能（feature）文件夹”为边界，按职责拆分子目录（命名可结合现有项目习惯微调，但语义必须一致）：

- `components/`：可复用组件
- `pages/` 或 `views/`：页面级组件
- `composables/`（Vue）或 `hooks/`（React）：可复用状态/副作用封装
- `services/`：API/请求封装
- `utils/`：纯工具函数
- `constants/`：静态常量/枚举
- `types/`：类型定义

抽离规则：

- 可复用的 JS/TS 逻辑尽量抽离到 `utils/` / `composables|hooks/` / `services/` 等，并补齐方法头注释。
- 每个方法尽量独立、低耦合、可测试；减少对隐式外部状态的依赖。

## UI/UX 与视觉实现（强制结合）

在任何前端任务中，必须结合：

- `/ui-ux-pro-max`：可用性、响应式、可访问性、交互反馈、信息架构等关键规则
- `/frontend-design`：风格方向、排版、色彩、空间、动效与细节质感的落地

使用深度由任务复杂度自行判断（从轻量规则约束到完整设计系统均可），但不得忽略可访问性与交互基本面。

## 输出交付物（强制）

每次交付必须包含：

- 变更摘要（目标/原因/取舍）
- 涉及文件清单（新增/修改/删除）
- 当新增较多时：文件树/目录结构
- 少量高信号关键代码片段或接口说明（避免大段堆砌）
- 当涉及重构或行为变化时：风险点、影响面与回滚建议

## 最小禁止项（强制）

- 禁止无关重构（与用户目标无关的结构调整一律不做）
- 禁止引入新依赖/新 UI 库（除非用户明确同意）
- 禁止破坏公共 API（对外组件 props、导出函数签名、对接接口契约等）
